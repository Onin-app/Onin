## 引言

Baize 是一个基于 Tauri 构建的桌面应用，支持通过插件扩展功能。本文档详细说明 Baize 插件的结构、配置方式、权限系统、命令定义以及加载机制。通过本指南，开发者可以快速理解如何创建和配置一个标准的 Baize 插件。

## 插件目录结构

一个标准的 Baize 插件应放置在用户数据目录下的 `plugins` 文件夹中，每个插件以独立的子目录存在。插件目录的基本结构如下：

```text
plugins/
└── my-plugin/
    ├── manifest.json
    ├── main.js 或 index.html
    ├── assets/
    │   ├── icon.png
    │   └── style.css
    └── lib/
        └── utils.js
```

- **插件根目录**：以插件 ID 命名的文件夹，位于 `app_data_dir()/plugins/` 下。
- **manifest.json**：插件的配置文件，定义插件元信息、入口、权限和命令。
- **入口文件**：由 `manifest.json` 中的 `entry` 字段指定，可以是 `.js`（无界面插件）或 `.html`（UI 插件）。
- **资源文件**：如图标、样式、脚本等，可通过相对路径引用。

## manifest.json 配置详解

`manifest.json` 是插件的核心配置文件，必须包含以下字段：

### 必需字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 插件唯一标识符，建议使用反向域名格式（如 `com.example.myplugin`） |
| `name` | string | 插件显示名称 |
| `version` | string | 插件版本号，遵循语义化版本规范 |
| `description` | string | 插件功能描述 |
| `entry` | string | 插件入口文件路径，相对于插件根目录 |

### 可选字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `type` | string | 插件类型，可选值为 `webview` 或 `headless`，默认根据 `entry` 扩展名自动判断 |
| `commands` | array | 插件支持的命令列表 |
| `permissions` | object | 插件所需权限声明，采用命名空间结构 |

## 权限系统详解

插件权限通过 `permissions` 字段声明，现已重构为包含 `http`、`storage`、`notification`、`command` 等子模块的命名空间结构。此变更旨在提供更直观、灵活且易于扩展的权限控制系统。

### http 权限

```json
"permissions": {
  "http": {
    "enable": true,
    "allowUrls": ["https://api.example.com", "http://localhost:3000"],
    "timeout": 30000,
    "maxRetries": 3
  }
}
```

- **作用**：允许插件发起网络请求到指定的域名。
- **核心字段**：
  - `enable`: 布尔值，是否启用 HTTP 权限。
  - `allowUrls`: 字符串数组，包含所有允许访问的域名或模式，支持通配符 `*.`。
  - `timeout`: 可选，请求超时时间（毫秒）。
  - `maxRetries`: 可选，最大重试次数。
- **安全影响**：
  - 插件只能访问在 `allowUrls` 数组中明确列出的域名。
  - 未声明的域名请求将被 Rust 后端的权限检查逻辑阻止。
  - 支持通配符 `*.example.com`，但不推荐用于根域。
- **权限检查流程**：
  1. 插件调用 `baize.request()` API。
  2. 请求通过 IPC 转发至 Rust 后端。
  3. `check_http_permission` 函数验证 `permissions.http.enable` 是否为 `true`。
  4. 使用 `is_url_allowed` 函数逐一匹配 `allowUrls` 规则。
  5. 匹配成功则执行请求，失败则返回 `PermissionDeniedError`。

## 命令系统详解

插件通过 `commands` 数组定义可被调用的功能命令。

### 命令定义结构

```json
"commands": [
  {
    "code": "search",
    "name": "搜索内容",
    "description": "在知识库中搜索相关信息",
    "keywords": [
      { "name": "搜索", "type": "prefix" },
      { "name": "find", "type": "prefix" }
    ]
  }
]
```

- `code`：命令内部标识符，用于程序调用。
- `name`：用户可见的命令名称。
- `description`：命令功能描述。
- `keywords`：触发命令的关键词，支持 `prefix` 类型（前缀匹配）。

### 命令执行流程

1. 用户输入关键词触发命令。
2. 主应用调用 `execute_plugin_command` API。
3. 根据插件类型（webview/headless）分发执行。
4. 插件返回结果通过 `plugin_command_result` 回调。

## 插件加载机制

Baize 在启动时自动扫描并加载插件。

### 加载流程

1. **扫描插件目录**：读取 `app_data_dir()/plugins/` 下所有子目录。
2. **读取 manifest.json**：验证文件存在并解析 JSON 内容。
3. **注册插件信息**：将插件元数据存入内存中的 `PluginStore`。
4. **按需加载执行**：
   - **Headless 插件**（`.js`）：在后台线程中通过 Deno 运行时执行。
   - **Webview 插件**（`.html`）：创建独立窗口加载 UI。

### 加载代码路径

```rust
pub fn load_plugins(
    app: tauri::AppHandle,
    store: State<PluginStore>,
) -> Result<Vec<LoadedPlugin>, String>
```

该函数遍历插件目录，读取每个插件的 `manifest.json`，并将其加入全局插件存储。

## 自定义协议 plugin://

Baize 使用自定义 URI 协议 `plugin://` 来安全地加载插件资源。

### 协议格式

```text
plugin://localhost/{plugin_dir_name}/{file_path}
```

例如：
```text
plugin://localhost/my-plugin/index.html
```

### 协议处理流程

1. **请求拦截**：Tauri 拦截所有 `plugin://` 开头的请求。
2. **路径解析**：提取插件目录名和文件路径。
3. **文件读取**：从本地插件目录读取对应文件。
4. **内容处理**：
   - HTML 文件：重写资源路径为相对路径（如 `/assets/` → `./assets/`）。
   - 其他文件：直接返回内容。
5. **设置响应头**：根据文件类型设置正确的 `Content-Type` 和 CORS 头。

### 安全特性

- **隔离访问**：插件只能通过协议访问自身目录下的文件。
- **CORS 支持**：响应头包含 `Access-Control-Allow-Origin: *`，允许插件内部跨域请求。
- **缓存控制**：设置 `Cache-Control: no-cache`，确保资源更新及时生效。

## 插件开发 SDK

Baize 提供了 `baize-sdk` 供插件开发者使用，封装了与主应用通信的底层细节。

### 核心功能

- **跨环境兼容**：通过 `getEnvironment()` 检测运行环境（`headless` 或 `webview`）。
- **IPC 通信**：统一的 `invoke` 和 `listen` 接口，适配不同环境。
- **API 封装**：提供 `request`、`notification`、`command`、`storage` 等高级 API。

### 命令处理器注册

插件通过 `registerCommandHandler` 注册命令处理器：

```ts
import { registerCommandHandler } from 'baize-sdk';

registerCommandHandler(async (command, args) => {
  if (command === 'search') {
    return await searchService(args.query);
  }
});
```

该函数内部使用 `dispatch` 根据环境选择正确的事件监听方式。

## 完整配置示例

```json
{
  "id": "com.baize.example",
  "name": "示例插件",
  "version": "1.0.0",
  "description": "一个演示 Baize 插件功能的示例",
  "entry": "index.html",
  "type": "webview",
  "permissions": {
    "http": {
      "enable": true,
      "allowUrls": ["https://api.example.com"]
    },
    "storage": {
      "enable": true,
      "local": true,
      "session": false
    },
    "notification": {
      "enable": true,
      "sound": true,
      "badge": false
    }
  },
  "commands": [
    {
      "code": "greet",
      "name": "打招呼",
      "description": "向用户问好",
      "keywords": [
        { "name": "你好", "type": "prefix" },
        { "name": "hello", "type": "prefix" }
      ]
    },
    {
      "code": "fetch",
      "name": "获取数据",
      "description": "从远程 API 获取数据",
      "keywords": [
        { "name": "获取", "type": "prefix" }
      ]
    }
  ]
}
```

### 字段说明

- `id`：插件唯一标识。
- `name`：用户界面显示名称。
- `entry`：入口 HTML 文件。
- `permissions`：声明 HTTP、存储和通知权限。
- `commands`：定义两个可触发命令。

## 总结

Baize 插件系统提供了一套完整、安全且灵活的扩展机制。通过标准的 `manifest.json` 配置，开发者可以定义插件的元信息、权限和功能命令。系统通过自定义 `plugin://` 协议实现资源的安全隔离加载，并利用 Tauri 的原生能力确保跨平台兼容性。配合 `baize-sdk`，插件可以轻松实现与主应用的双向通信，构建丰富的扩展功能。本次权限系统的重构显著提升了配置的灵活性和安全性，为未来功能扩展奠定了坚实基础。
